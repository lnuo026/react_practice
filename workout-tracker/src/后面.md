## 一、大局观：这一章在做什么？

你已经会画 ER 图（设计蓝图）。这一章要做的事是：

```
ER 图（概念设计）  →  关系数据库表（实际可用的数据库）
```

这个翻译过程叫做 **ER-to-Relational Mapping（映射）**，共分 7 个步骤，按顺序执行。

---

## 二、映射的三个目标

| 目标           | 说明                                        |
| -------------- | ------------------------------------------- |
| 保留所有信息   | 所有属性都不能丢失                          |
| 尽量保留约束   | 关系模型无法保留所有约束（如精确比例 1:10） |
| 最小化 NULL 值 | 减少空值，节省空间，提高数据质量            |

## 三、前置知识（必须先懂）

### 3.1 表（Relation / Table）

数据库用表格存数据，类似 Excel：

```
EMPLOYEE 表
┌────────┬───────┬──────────┬───────┐
│ Fname  │ Lname │   Ssn    │  ... │
├────────┼───────┼──────────┼───────┤
│  John  │ Smith │ 123-45-6 │  ... │
│  Mary  │ Wong  │ 789-01-2 │  ... │
└────────┴───────┴──────────┴───────┘
行 = 一条记录    列 = 一个属性
```

### 3.2 主键（Primary Key）

- 能**唯一识别每一行**的列（或列的组合）
- 不能重复，不能为空
- ER图中用**下划线**标注，如 `Ssn`

### 3.3 外键（Foreign Key）

- **引用另一张表主键**的列
- 用来建立两张表之间的联系
- 外键的值必须在被引用的表中存在（或为NULL）

```
DEPARTMENT 表                  EMPLOYEE 表
┌──────────┬────────┐         ┌─────┬──────┐
│ Dnumber  │ Dname  │ ←────── │ Dno │ ...  │
│    5     │Research│         │  5  │ ...  │
└──────────┴────────┘         └─────┴──────┘
    主键                          外键（指向 Dnumber）
```

### 3.4 基数（Cardinality）回顾

| 类型 | 含义   | 生活例子             |
| ---- | ------ | -------------------- |
| 1:1  | 一对一 | 一个部门只有一个经理 |
| 1:N  | 一对多 | 一个部门有多个员工   |
| M:N  | 多对多 | 一个员工参与多个项目 |

### 3.5 全参与 vs 部分参与

| 类型                | ER图符号   | 含义                     |
| ------------------- | ---------- | ------------------------ |
| 全参与（Total）     | 双线 `═══` | 每个实例都必须参与该关系 |
| 部分参与（Partial） | 单线 `───` | 可以不参与该关系         |

例：

- DEPARTMENT **全参与** MANAGES → 每个部门必须有经理
- EMPLOYEE **部分参与** MANAGES → 不是每个员工都是经理

### 3.6 弱实体（Weak Entity）

- **不能独立存在**，必须依附于"所有者实体"
- ER图用**双线矩形**表示
- 弱实体的**部分键（Partial Key）**用**双下划线椭圆**表示
- 弱实体主键 = 所有者主键 + 部分键

### 3.7 多值属性（Multivalued Attribute）

- 一个实体的某属性可以有**多个值**
- ER图用**双线椭圆**表示
- 例：DEPARTMENT 的 Locations（一个部门可在多个地点）

### 3.8 派生属性（Derived Attribute）

- 可以由其他属性**计算得出**，不需要直接存储
- ER图用**虚线椭圆**表示
- 例：Number_of_employees（可通过统计员工数量得出）
- **映射时：派生属性不存入表中**

### Step 1：普通实体类型映射

**规则：**

1. 对 ER 图中每一个**普通（强）实体类型 E**，创建一张关系表 R
2. 把 E 的所有**简单属性**作为 R 的列
3. **复合属性** → 拆开为各个简单子属性（不保留复合属性本身）
4. **派生属性** → 不放入表中
5. 从 E 的键属性中选一个作为 R 的**主键**
6. 若键属性本身是复合的，则其所有简单子属性共同构成主键

**COMPANY 例子：**

```
EMPLOYEE
┌────────┬───────┬───────┬──────────┬────────┬─────────┬─────┬────────┐
│ Fname  │ Minit │ Lname │   Ssn    │ Bdate  │ Address │ Sex │ Salary │
└────────┴───────┴───────┴──────────┴────────┴─────────┴─────┴────────┘
                              ↑主键
（Name 是复合属性，拆成 Fname + Minit + Lname）

DEPARTMENT
┌─────────┬──────────┐
│  Dname  │ Dnumber  │
└─────────┴──────────┘
               ↑主键
（Locations 是多值属性，Step 6 处理）
（Number_of_employees 是派生属性，不放入）

PROJECT
┌─────────┬──────────┬──────────┐
│  Pname  │ Pnumber  │ Plocation│
└─────────┴──────────┴──────────┘
               ↑主键
```

---

### Step 2：弱实体类型映射

**规则：**

1. 对每个**弱实体类型 W**（所有者实体为 E），创建关系表 R
2. 包含 W 的所有简单属性（复合属性拆开）
3. 将所有者实体 E 对应表的**主键引入 R，作为外键**
4. R 的主键 = **所有者主键 + W 的部分键**（若 W 无部分键，则主键仅为所有者主键）

**COMPANY 例子：**

DEPENDENT 是弱实体，所有者是 EMPLOYEE：

- 引入 EMPLOYEE.Ssn 作为外键，改名为 Essn
- 部分键 = Dependent_name（同一员工名下家属名不重复）
- 主键 = {Essn, Dependent_name}

```
DEPENDENT
┌──────┬────────────────┬─────┬───────┬──────────────┐
│ Essn │ Dependent_name │ Sex │ Bdate │ Relationship │
└──────┴────────────────┴─────┴───────┴──────────────┘
  ↑外键    ↑部分键
  └────────────────┘
      联合主键
```

---

### Step 3：二元 1:1 关系类型映射

对每个二元 1:1 关系 R，找到参与的两张表 S 和 T。

有**三种方法**可选：

#### 方法1：外键法（2张表）—— 最常用

- 选择一张表（建议选**全参与**的那边，记为 S），将另一张表的主键引入 S 作为外键
- 选全参与的一边原因：避免大量 NULL 值出现
- 关系本身的属性也加入 S

**COMPANY 例子（MANAGES 关系）：**

- DEPARTMENT 全参与（每个部门必须有经理）→ 选 DEPARTMENT 作为 S
- EMPLOYEE 部分参与（不是每个员工都是经理）
- 把 EMPLOYEE.Ssn 放入 DEPARTMENT 表，改名 Mgr_ssn
- 关系属性 Start_date 也放入 DEPARTMENT 表，改名 Mgr_start_date

```
DEPARTMENT（更新后）
┌─────────┬──────────┬─────────┬───────────────┐
│  Dname  │ Dnumber  │ Mgr_ssn │ Mgr_start_date│
└─────────┴──────────┴─────────┴───────────────┘
               ↑主键    ↑外键→Employee.Ssn
```

若选 EMPLOYEE 放外键则不合适：不是每个员工都是经理，Dnumber 对大多数员工为 NULL，浪费空间。

#### 方法2：合并法（1张表）

- 将两个实体和关系合并成一张大表
- **适用条件**：两边都是全参与时可以考虑（但并非强制要求）

#### 方法3：关系表法（3张表）

- 新建第三张表，专门存储两边主键的对应关系
- 1:1 情况下很少使用，但在需要交叉引用时可选

---

### Step 4：二元 1:N 关系类型映射

**规则：**

1. 对每个二元 1:N 关系 R，找到 **N 端**对应的关系表 S
2. 将 **1 端**对应表 T 的主键，加入 S 作为外键
3. 关系本身的**简单属性**也加入 S

**记忆口诀：外键放 N 端**

**COMPANY 例子：**

**WORKS_FOR（N个员工属于1个部门）：**

- N 端 = EMPLOYEE，1 端 = DEPARTMENT
- 把 DEPARTMENT.Dnumber 放入 EMPLOYEE，改名 Dno

**CONTROLS（1个部门控制N个项目）：**

- N 端 = PROJECT，1 端 = DEPARTMENT
- 把 DEPARTMENT.Dnumber 放入 PROJECT，改名 Dnum

**SUPERVISION（1个员工监督N个员工，自身关系）：**

- N 端 = 被监督的员工，1 端 = 监督者
- 把 EMPLOYEE.Ssn（监督者）放入 EMPLOYEE 表，改名 Super_ssn
- Super_ssn 是外键，引用的是同一张表的 Ssn（自引用）

```
EMPLOYEE（更新后）
┌───────┬───────┬───────┬─────┬───────┬─────────┬─────┬────────┬───────────┬─────┐
│ Fname │ Minit │ Lname │ Ssn │ Bdate │ Address │ Sex │ Salary │ Super_ssn │ Dno │
└───────┴───────┴───────┴─────┴───────┴─────────┴─────┴────────┴───────────┴─────┘
                              ↑主键                               ↑自引用外键  ↑外键

PROJECT（更新后）
┌─────────┬──────────┬──────────┬──────┐
│  Pname  │ Pnumber  │ Plocation│ Dnum │
└─────────┴──────────┴──────────┴──────┘
               ↑主键               ↑外键→Department.Dnumber
```

---

### Step 5：二元 M:N 关系类型映射

**规则：**

1. 对每个二元 M:N 关系 R，**新建一张关系表 S**
2. 将两端实体的主键都引入 S 作为外键
3. **S 的主键 = 两个外键的组合**
4. 关系本身的简单属性（或复合属性拆开的简单分量）也加入 S

**注意：M:N 关系必须新建表，不能用外键法！**（因为无论放哪一边都会产生重复或不一致）

**COMPANY 例子（WORKS_ON 关系）：**

- M个员工参与N个项目
- 新建 WORKS_ON 表
- 引入 EMPLOYEE.Ssn → 改名 Essn，PROJECT.Pnumber → 改名 Pno
- 关系属性 Hours 也放入
- 主键 = {Essn, Pno}

```
WORKS_ON
┌──────┬─────┬───────┐
│ Essn │ Pno │ Hours │
└──────┴─────┴───────┘
  ↑外键  ↑外键
  └──────────┘
    联合主键
```

含义：员工 Essn 在项目 Pno 上工作了 Hours 小时。

---

### Step 6：多值属性映射

**规则：**

1. 对每个多值属性 A，**新建一张关系表 R**
2. R 包含：多值属性 A 本身 + 原实体的主键 K（作为外键）
3. R 的主键 = {A, K} 的组合
4. 若 A 是复合多值属性，则拆开为简单分量

```
：多值属性映射——从零理解

  ---
  第一步：什么是多值属性？

  普通属性：一个员工只有一个出生日期 → 单值属性
  多值属性：一个部门可以有多个办公地点 → 多值属性

  在 ER 图里，多值属性用双椭圆表示。

  ---
  第二步：为什么不能直接放在原表里？

  假设 DEPARTMENT 表长这样：

  ┌──────────┬─────────┬───────────┐
  │ DeptName │ DNumber │ Location  │
  ├──────────┼─────────┼───────────┤
  │ Research │ 5       │ Houston   │
  ├──────────┼─────────┼───────────┤
  │ Research │ 5       │ Stafford  │
  ├──────────┼─────────┼───────────┤
  │ Research │ 5       │ Sugarland │
  └──────────┴─────────┴───────────┘

  问题： 主键 DNumber 重复出现，违反关系数据库规则（每行必须唯一）。

  解决方法： 把多值属性单独拆出来建一张新表。

  ---
  第三步：规则拆解

  原来的情况：

  DEPARTMENT 实体
  ├── DeptName
  ├── DNumber（主键）
  └── Locations ← 多值属性（双椭圆）

  映射后新建一张表 R：

  DEPT_LOCATIONS 表
  ├── Dnumber（外键，来自 DEPARTMENT 的主键）
  └── Dlocation（多值属性本身）

  这张新表的主键 = {Dnumber + Dlocation} 组合
```

**COMPANY 例子（DEPARTMENT 的 Locations 属性）：**

```
DEPT_LOCATIONS
┌──────────┬──────────┐
│ Dnumber  │ Dlocation│
└──────────┴──────────┘
  ↑外键→Department.Dnumber
  └──────────────────┘
        联合主键

数据示例：
│ 5 │ Houston  │  ← 部门5在两个地点
│ 5 │ Stafford │
│ 4 │ Bellaire │
```

---

### Step 7：N 元关系类型映射（n > 2）

**规则：**

1. 对每个 n 元关系 R（n > 2，即超过2个实体参与），**新建一张关系表 S**
2. 将所有参与实体的主键引入 S 作为外键
3. 关系本身的简单属性也加入 S

**COMPANY 例子（SUPPLY 三元关系）：**

参与实体：SUPPLIER、PART、PROJECT
含义：某供应商向某项目提供某零件若干数量

```
SUPPLY
┌───────┬───────────┬─────────┬──────────┐
│ Sname │ Proj_name │ Part_no │ Quantity │
└───────┴───────────┴─────────┴──────────┘
  ↑外键     ↑外键      ↑外键
  └──────────────────────┘
           联合主键
```

## 六、COMPANY 数据库最终映射结果

经过 7 步映射后，COMPANY ER 图变成以下 6 张表：

```
EMPLOYEE
Fname | Minit | Lname | Ssn(PK) | Bdate | Address | Sex | Salary | Super_ssn(FK) | Dno(FK)

DEPARTMENT
Dname | Dnumber(PK) | Mgr_ssn(FK) | Mgr_start_date

DEPT_LOCATIONS
Dnumber(PK,FK) | Dlocation(PK)

PROJECT
Pname | Pnumber(PK) | Plocation | Dnum(FK)

WORKS_ON
Essn(PK,FK) | Pno(PK,FK) | Hours

DEPENDENT
Essn(PK,FK) | Dependent_name(PK) | Sex | Bdate | Relationship
```

外键引用关系：

- EMPLOYEE.Dno → DEPARTMENT.Dnumber
- EMPLOYEE.Super_ssn → EMPLOYEE.Ssn（自引用）
- DEPARTMENT.Mgr_ssn → EMPLOYEE.Ssn
- DEPT_LOCATIONS.Dnumber → DEPARTMENT.Dnumber
- PROJECT.Dnum → DEPARTMENT.Dnumber
- WORKS_ON.Essn → EMPLOYEE.Ssn
- WORKS_ON.Pno → PROJECT.Pnumber
- DEPENDENT.Essn → EMPLOYEE.Ssn

## 七、ER 模型 vs 关系模型 对照表（PPT Table 9.1）

| ER 模型                 | 关系模型                  |
| ----------------------- | ------------------------- |
| 实体类型（Entity type） | 实体表（Entity relation） |
| 1:1 或 1:N 关系类型     | 外键（或关系表）          |
| M:N 关系类型            | **关系表** + 两个外键     |
| N 元关系类型            | **关系表** + N 个外键     |
| 简单属性                | 普通列（Attribute）       |
| 复合属性                | 拆成多个简单列            |
| 多值属性                | **新表** + 外键           |
| Value set（值域）       | Domain（域）              |
| 键属性                  | 主键（或候选键）          |

## 八、7步总口诀（记忆用）

```
Step 1  普通实体  → 直接建表，简单属性做列，选主键
Step 2  弱实体    → 建表 + 引入所有者主键（外键），联合主键
Step 3  1:1关系   → 外键法（优先放全参与那边）/ 合并法 / 关系表法
Step 4  1:N关系   → 外键放 N 端，1端主键引过去
Step 5  M:N关系   → 必须新建关系表，两个外键组合成主键
Step 6  多值属性  → 必须新建表，与原实体主键联合成主键
Step 7  N元关系   → 新建关系表，N个外键联合成主键
```


 ---                                                         
  问题1：这7步在做什么？
                                                              
  一句话：把 ER 图"翻译"成真正可以建数据库的表格。
                                                              
  ER 图（设计图纸）  →  7步映射  →  关系模式（实际数据库表）
                                                              
  就像建房子：                                                
  - ER 图 = 建筑设计蓝图（矩形、菱形、椭圆）                  
  - 关系模式 = 实际施工图纸（一张张表格）                     
                                                             
  ER 图只是"概念设计"，数据库不认识矩形和菱形，只认识表格+列+ 
  主键+外键。                                                 
                                                              
  ---                                                         
  问题2：为什么两张图长得不一样？                            
                                                              
  因为它们是同一个信息的两种不同表达方式：
                                                              
  ┌──────────────┬──────────────────┬──────────────────────┐  
  │              │      ER 图       │ 关系模式图（箭头图） │  
  ├──────────────┼──────────────────┼──────────────────────┤  
  │ 实体         │ 矩形             │ 变成一张表格         │ 
  ├──────────────┼──────────────────┼──────────────────────┤ 
  │ 关系（菱形） │ 菱形连接两个矩形 │ 消失了！ 变成 FK     │  
  │              │                  │ 列或新表             │  
  ├──────────────┼──────────────────┼──────────────────────┤  
  │ 属性         │ 椭圆             │ 变成表格的列         │  
  ├──────────────┼──────────────────┼──────────────────────┤ 
  │ 箭头含义     │ 无箭头           │ FK → PK              │ 
  └──────────────┴──────────────────┴──────────────────────┘  
  
  ---                                                         
  具体例子：用 MOVIES 说明
                                                              
  ER 图里长这样：
  [Employee] ──works_for──→ [Company]                         
      N                          1   
  菱形 works_for 连接两个矩形。                               
                                                              
  变成关系模式后：                                            
  菱形 works_for 消失了！                                     
  变成 EMPLOYEE 表里多一列 Reg_ID（FK）                      
                                                              
  EMPLOYEE                                                    
  ┌─────┬──────┬────────┬────────────┬────────┐               
  │ Ssn │ Name │ Gender │ Birth_date │ Reg_ID                 
  │←这列就是works_for关系                                     
  └─────┴──────┴────────┴────────────┴────────┘               
                                        FK↓                   
                                     COMPANY                  
                                                             
  ---                                                         
  7步的本质就是：把菱形变掉
                                                              
  ┌────────────────────┬───────────────────────────────────┐
  │   ER 图里的东西    │          7步之后变成什么          │  
  ├────────────────────┼───────────────────────────────────┤  
  │ 普通实体（矩形）   │ 直接建一张表（Step 1）            │  
  ├────────────────────┼───────────────────────────────────┤  
  │ 弱实体（双矩形）   │ 建表 + 引入所有者主键（Step 2）   │  
  ├────────────────────┼───────────────────────────────────┤  
  │ 1:1 关系（菱形）   │ 消失，变成某张表里的一列 FK（Step │  
  │                    │  3）                              │  
  ├────────────────────┼───────────────────────────────────┤ 
  │ 1:N 关系（菱形）   │ 消失，变成 N 侧表里的一列         │  
  │                    │ FK（Step 4）                      │  
  ├────────────────────┼───────────────────────────────────┤ 
  │ M:N 关系（菱形）   │ 消失，变成一张全新的表（Step 5）  │  
  ├────────────────────┼───────────────────────────────────┤  
  │ 多值属性（双椭圆） │ 消失，变成一张全新的表（Step 6）  │ 
  ├────────────────────┼───────────────────────────────────┤  
  │ N 元关系（菱形）   │ 消失，变成一张全新的表（Step 7）  │
  └────────────────────┴───────────────────────────────────┘  
                  
  ---                                                         
  总结            
                                                             
  两张图看起来不一样，是因为：
                                                              
  ▎ ER 图里的菱形（关系）在关系模式里全部消失了，             
  ▎ 它们要么变成某张表里的外键列，要么变成一张新表。          
                                                              
  这就是为什么箭头图里没有菱形，只有表格和箭头。

## 九、常见易错点

| 错误                     | 正确做法                            |
| ------------------------ | ----------------------------------- |
| M:N关系用外键法          | M:N必须新建关系表                   |
| 多值属性直接放在实体表里 | 多值属性必须新建单独的表            |
| 派生属性也存入表         | 派生属性不存入表                    |
| 复合属性整体存入         | 复合属性拆开为简单属性              |
| 1:1外键放部分参与那边    | 1:1外键优先放全参与那边（减少NULL） |
| 弱实体主键只用部分键     | 弱实体主键 = 所有者主键 + 部分键    |
